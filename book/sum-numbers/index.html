    Sum

    Tom: "Miss Sophia wants us to sum 0 to 5"
    Me: "Oh yeah, 0 + 1 = 1; 1 + 2 = 3; ..."
    Tom: "Wait, wait, i know the sum from 0 to 4.
    I remember it's 10."
    Me: "Okay, in that case, the answer is 15 then!"

    Say we want to find out the sum of a list of number from 0 to n, we start to write an algorithm.

    function sum(n) {
  return M + n;
}

    What is M ?

    Me: "Did you forget something? what is M?"
    Tom: "oh, I forgot. M is sum(n - 1)!"
    Me: "What?"
    Tom: "What's the matter?"
    Me: "... Nothing."

    Replacing M with sum(n-1), like someone knew sum(n-1) already.

    function sum(n) {
  return sum(n - 1) + n;
}

    Me: "I don't really think this would work. "
    Tom: "If I know sum(4), I can tell you sum(5)."
    Me: "How could I know sum(4) first."
    Tom: "Well, if you can tell me what is sum(3)."
    Me: "How am I supposed to tell you sum(3)?"
    Tom: "True, but I think I know what is sum(0)".
    Me & Tom: "That is zero!"
    Tom: "then I know sum(1) and ..."
    Me & Tom: "then I know sum(5)."

    Adding sum(0) = 0 that we know from the beginning.

    function sum(n) {
  if (n == 0) return 0;
  return sum(n - 1) + n;
}

    Run it.

    > count(5)
15

    It's 15, just great!

    Me: "I didn't expect we can get it this far."
    Tom: "Me either, so what's this?"
    Me & Tom: "Recursion?!"

    Recursion it is.

    Function instance

    What is a recursion?

    function sum(n) {
  if (n == 0) return 0;
  return sum(n - 1) + n;
}

    In the code, it's a function named sum, inside it calls a function named sum. So it's a function calling itself.

    When a function keeps calling itself, each one that is launched is called a function instance, such as sum(n) and sum(n-1). Each function instance can be only distinguished by the argument passed to the function, such as n and n-1. The argument essentially becomes the name of the function instance.

    What is the difference between each function instance, say between sum(n) and sum(n-1)? It’s subtle. They follow the same logic from the same function body. On the other hand, each instance carries slight variations due to conditional statement inside the function body. In our sum case, sum(5) is quite similar to sum(4), but different from sum(0). Since the function uses a conditional if statement for sum(0).

    We call sum(0) instance as a boundary. When the code reaches there, it has to go back quickly to where it was called. It behaves almost as if it's not called. A instance serving as a boundary is equally as important as other instances. Without boundary, the problem is unbounded, thus un-solvable.

    Any function’s job is to run through some calculations and return the result. During the visit of the function instance, it maintain its active status. The total number of the active instances fluctuates and could reach any number during the running of the algorithm.

    The function instance and the boundary are two key elements of a recursion. We’ll come back to them often throughout this book.

    Recursion the right way

    One of the challenge of learning recursion is that we are very used to the classical way of doing things using a for loop. When it comes to the recursion, things seem to be quite unmanageable. Suddenly we seem to run out of steam in controlling things precisely in coding.

    I don't want to go through the recursion analysis in most of textbooks or online materials, which is time-consuming and mind-blogging. Because I don't want you to think recursion is hard, especially if you are new to this subject. First impression is important, and I don't want to leave you with bad one.

    I'll introduce you the right way of looking at recursion. We'll go from a natural way of how we come to the recursion at the very first place. In a sense, I want you to relax a bit and lear something new. If you happen to know the classical way, please pretend you don't know it for the first few chapter.

    Tom: “We should read the function line by line.”
    Me: “Isn’t this what we just did?”
    Tom: “No, I didn’t mean by the compiler.”
    Me: “I’m confused, what you mean?”
    Tom: “I mean by us.”

    Let’s walk through the recursive function we wrote once again. What does the function do after we call it?

    function sum(n) {
  if (n == 0) return 0;
  return sum(n - 1) + n;
}

    Say we start at one function instance n.

    sum(n) -> sum(n - 1) + n

    Quickly it asks us to call sum(n-1) and then the result to n.

    sum(n) -> abc(n - 1) + n

    Pretend we replace the sum(n-1) by abc(n-1) for a second. What this abc function does is to return us the result of sum(n-1). This is the case when n is not 0.

    sum(n) -> 0

    When n is 0, we return the number 0 right away.

    Now let’s organize our thoughts a bit.

    Tom: “How do you feel?”
    Me: “I don’t know. Aren’t they same?”
    Tom: “Have you gone through every line?”
    Me: “Yes, but I felt I did little.”
    Tom: “Nice, isn’t it?”

    We walk you through the logic of the code as we normally do to any piece of computer code. The thing we don't want you do is digging into each instance function. Because we believe that work is the compiler's job.

    We don't go through the detail of each function instance, instead we take for granted what's returned from it, and move on to the next line of the code.

    Each of us only has one brain, we just don't have enough brain power to absorb the details of all function instances at the same time.

    For instance, when I ask you to trace through 5 function calls, you find it inspiring and encouraging; if I raise the number to 23, you might start to find it tedious but still be proud afterwards; now if I raise it to 178, you’d give up in no time. Don't you think we should develop a more consistent way of reasoning when things scale up?

    Therefore this book strongly suggests we don't look at the recursion as how compiler looks at it. Instead, we approach the problem as natural as possible. And no matter how large the number is, we stay calm and focus more on the intrinsics of the problem.

    Running time measurement

    In this book, we use recursion to write computer algorithm. Any algorithm book would tell you how to measure algorithm's success on the front.

    This is no exception here.

    In computer science, one of the key metric of a computer algorithm, is its running time, expressed by the notation O. We call it the big O. To name a few popular running time expressions.

    O(n)
O(n^2)
O(2^n)
O(log(n))
O(n!)

    The number n corresponds to the size of the problem. Taking O(n) as an example, it means the running time is in linear relationship against n, when n increases, it increases the running time linearly.

    Notice there's no coefficient or multiple terms in any of these expressions. Because the order of running time has no interest at the difference between O(n), O(2n), or O(n+1), because to the running time, all increase linearly with n. Big O only tracks the quickest growing order.

    Another way to look at the big O is that the unit time for each problem varies. This problem might be doing addition operation, while another problem might be doing multiplication. Even though we know multiplication is slower than addition, we don't normally pay much attention to this level of detail. Instead we want to know how many times each operation has been performed.

    Given couple of problems with the following order, can we figure out which one runs faster?

    O(1)
O(n)
O(n^2)

    We actually don't know. Because the order of running time does not measure the exact time. We can't be sure whether the unit time for each problem is the same. However, for the same problem, when there's one unit time, if one algorithm gives O(n) and another one gives O(n^2), then we know the former should be more performant. Because the linear form of running time grows slower than the quadratic form of running time when the problem scales up with increasing n.

    Ok, now we know what is the big O, let's think about what’s the running time for the sum algorithm we wrote?

    Say the time spent on running each function instance the unit time, considering it basically does the addition of two numbers, we can count how many times a function instance is called.

     start
  ---
   n
  n-1
   .
   .
   .
   1
   0
  ---
boundary

    To know how many instances are visited, imagine we start from the first function instance n, and then move into n-1, and continues until it reaches 0 which is our boundary. It's in total n+1 instances.

    Thus the running time of our algorithm is said to be O(n), meaning the running time is linearly proportional to n. If sum(1) takes 1 microsecond in a real run, then we expect sum(5) should spend approximately around 5 microseconds. Of course 5 microseconds isn’t an accurate measurement, but we don’t expect it to take more than 10 microseconds, because that would break the O(n) analysis.

    Run it!

    We have the algorithm written, and we know how fast it runs against n, let's run it.

    > sum(5)
15

> sum(10)
55

> sum(100)
5050

> sum(1000)
500500

    Great! Don't you know the sum takes shape of a pattern 5x5x for each magnitude of n. Guess what is sum(10000)?

    > sum(1000)
500500

    Yes, it's 50005000.
